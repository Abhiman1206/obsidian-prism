<risks>
- Platform defaults (timeouts, regions, or build images) may differ from local assumptions and affect behavior.
- Missing or mis-scoped environment variables can cause runtime failures after deployment.
- Cross-service release order mistakes can create temporary downtime or broken frontend calls.
- Build cache or dependency drift between local and hosted build images can introduce non-reproducible deploy failures.
</risks>

<assumptions>
- Organization accounts and project access are available on Vercel and Render.
- DNS/domain requirements are either optional or handled outside this phase.
- Phase 7 integration is stable enough to deploy without additional contract changes.
</assumptions>

<rollback>
- Roll backend to the prior Render service version before frontend rollback if API regressions are detected.
- Roll frontend on Vercel to the previous production deployment if UI release is incompatible.
- Maintain versioned environment snapshots so rollback restores both code and configuration state.
- If deployment fails pre-smoke, halt frontend promotion and keep prior backend/frontend pair active to avoid split-version incompatibility.
</rollback>

- Frontend deploys on Vercel using environment-driven backend URL configuration.

- Deployment and rollback runbook is executable and complete for operational handoff.
